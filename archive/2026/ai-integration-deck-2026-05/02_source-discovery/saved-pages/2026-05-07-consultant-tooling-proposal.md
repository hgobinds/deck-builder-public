## Key Thesis (direct quote)

> "The goal is not to automate the consultant out of the process. The goal is to remove repeated setup, forgotten context, inconsistent review, and weak handoff so the consultant can spend more time on judgment, diagnosis, and client trust."

---

## Consultant Operating Loop

```
New lead / client request
→ Intake + Discovery
→ Scope + Proposal
→ Delivery Workspace
→ Review + Client Comms
→ Handoff + Case Study
```

---

## Three Pillars of Consultant Work

| Pillar | Activities |
|---|---|
| Client Setup | intake, discovery, scope, proposal |
| Delivery Work | deliverables, QA review, client comms, handoff |
| Practice Growth | case studies, follow-up, pipeline memory, reusable proof |

---

## 15 Proposed AI Tools (Mapped to Consultant Workflow)

**Phase 1 — Core Delivery (highest priority)**
1. Client Workspace Generator
2. Intake Brief Builder
3. Scope Agreement Builder
4. Meeting Notes Processor
5. Review Gate / Quality Checker
6. Client Communication Drafter
7. Handoff Package Builder

**Phase 2 — Control + Memory**
8. Change Request Detector
9. Workspace Auditor
10. Decision Log Builder
11. Consultant Command Center

**Phase 3 — Growth**
12. Proposal / SOW Builder
13. Case Study Anonymizer
14. Deliverable Builder (slide deck, memo, report, roadmap, etc.)
15. Practice analytics or pipeline reporting

---

## Design Principle

> "Each tool should do one job in the workflow. The consultant should not need to remember where everything goes."

Flow: `messy client input → [tool reads context] → structured artifact → saved in correct workspace folder`

---

## Success Criteria (from document)

- New client workspace created in minutes
- Every engagement has a structured intake brief
- No delivery starts without scope and acceptance criteria
- Meeting notes become decisions and action items automatically
- Review happens before client delivery
- Handoff includes documentation, not just files
- Consultant can open one status file and know what needs attention
